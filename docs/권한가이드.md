# 권한 가이드

본 문서는 현재 시스템에서 사용하는 권한 정책을 운영 관점에서 이해하기 쉽게 정리한 문서입니다.

## 1. 권한 운영 원칙

현재 권한 정책은 아래 원칙으로 운영됩니다.

| 구분 | 원칙 |
|---|---|
| 직원 목록 | 로그인한 사용자는 누구나 조회 가능 |
| 직원 상세 | 별도 조회 권한이 있어야 조회 가능 |
| 엑셀 다운로드 | 별도 조회 권한이 있어야 다운로드 가능 |
| 직원 변경 | 별도 쓰기 권한이 있어야 가능 |
| 범위 제한 | 권한이 있어도 허용된 범위 내의 직원만 조회/변경 가능 |

쉽게 말하면, `직원 명단은 넓게 공개`, `직원 상세와 변경 작업은 엄격하게 통제`하는 구조입니다.

---

## 2. 현재 사용 중인 권한 코드

| 권한 코드 | 이름 | 설명 | 현재 상태 |
|---|---|---|---|
| `employee.read` | 직원 상세 조회 및 엑셀 다운로드 | 직원 상세 정보 조회와 범위 기반 엑셀 다운로드를 허용하는 권한 | 적용 중 |
| `employee.write` | 직원 생성 및 변경 관리 | 직원 생성, 프로필 수정, 상태 변경, 삭제, 엑셀 업로드를 허용하는 권한 | 적용 중 |
| `permission.group.manage` | 권한 그룹 관리 | 권한 그룹과 그룹별 계정 할당을 조회하고 관리할 수 있는 권한 | 적용 중 |
| `project.read` | 프로젝트 목록 및 상세 조회 | 프로젝트 목록, 상세 정보, 관련 조회 화면 접근을 허용하는 권한 | 시드만 반영 |
| `dashboard.read` | 대시보드 지표 조회 | 대시보드의 핵심 지표와 요약 정보를 조회할 수 있는 권한 | 시드만 반영 |

`project.read`, `dashboard.read`는 권한 코드와 기본 부여 정책은 들어가 있지만, 직원 권한처럼 본격적인 API 보호는 아직 정리 중입니다.

## 3. 기능별 권한 적용 현황

### 3-1. 직원 조회 기능

| 기능 | 로그인만 필요 | `employee.read` 필요 | 범위(scope) 적용 | 비고 |
|---|---|---|---|---|
| 직원 목록 조회 | O | X | X | 공개 가능한 직원 디렉터리 |
| 부서별 직원 목록 조회 | O | X | X | 부서 화면에서 사용 |
| 직원 상세 조회 | X | O | O | 범위 밖이면 `403` |
| 직원 엑셀 다운로드 | X | O | O | 허용 범위 직원만 다운로드 |
| 직급/등급/유형/상태/아바타 목록 조회 | O | X | X | 화면 입력값용 공통 목록 |

### 3-2. 직원 변경 기능

| 기능 | 로그인만 필요 | `employee.write` 필요 | 범위(scope) 적용 | 비고 |
|---|---|---|---|---|
| 직원 생성 | X | O | O | 요청한 부서 기준 검사 |
| 직원 수정 | X | O | O | 대상 직원과 변경 부서 모두 검사 |
| 직원 삭제 | X | O | O | 대상 직원 기준 검사 |
| 삭제 복구 | X | O | O | 삭제된 직원도 범위 검사 후 복구 |
| 퇴사 처리 | X | O | O | 대상 직원 기준 검사 |
| 휴직 처리 | X | O | O | 대상 직원 기준 검사 |
| 복직 처리 | X | O | O | 대상 직원 기준 검사 |
| 승진 처리 | X | O | O | 대상 직원 기준 검사 |
| 직원 엑셀 업로드 | X | O | O | 업로드 행 중 하나라도 범위 밖이면 전체 거부 |

### 3-3. 시스템 관리 기능

| 기능 | 로그인만 필요 | 별도 권한 필요 | 범위(scope) 적용 | 비고 |
|---|---|---|---|---|
| 권한 그룹 목록 조회 | X | `permission.group.manage` | X | 최고 관리자 전용 |
| 권한 그룹 상세 조회 | X | `permission.group.manage` | X | 최고 관리자 전용 |
| 커스텀 권한 그룹 생성/수정/삭제 | X | `permission.group.manage` | X | 시스템 그룹은 읽기 전용 |
| 권한 그룹에 계정 할당/해제 | X | `permission.group.manage` | X | 시스템 그룹도 계정 할당은 가능 |

## 4. 범위(scope) 의미

권한에는 범위가 함께 부여됩니다.

| scope | 의미 | 조회 권한에서의 의미 | 쓰기 권한에서의 의미 |
|---|---|---|---|
| `ALL` | 전체 | 모든 직원 조회 가능 | 모든 직원 관리 가능 |
| `OWN_DEPARTMENT` | 본인 부서 | 본인 부서 직원만 조회 가능 | 본인 부서 직원만 관리 가능 |
| `OWN_DEPARTMENT_TREE` | 본인 부서 + 하위 부서 | 하위 부서 포함 조회 가능 | 하위 부서 포함 관리 가능 |
| `CURRENT_PARTICIPATION` | 본인 현재 참여 | 현재 구현에서는 직원 조회 범위를 넓히지 않음 | 현재 구현에서는 직원 쓰기 범위를 넓히지 않음 |
| `SELF` | 본인 | 내 상세만 조회 가능 | 내 정보 일부만 수정 가능 |

---

## 5. `SELF` 권한의 실제 의미

`SELF`는 가장 제한적인 권한입니다.

### 5-1. `employee.read: SELF`

| 가능 | 불가능 |
|---|---|
| 본인 직원 상세 조회 | 다른 직원 상세 조회 |
| 본인 범위의 조회 응답 확인 | 다른 직원 엑셀 다운로드 |

### 5-2. `employee.write: SELF`

| 가능 | 불가능 |
|---|---|
| 본인 이름 수정 | 직원 생성 |
| 본인 생년월일 수정 | 직원 삭제 |
| 본인 아바타 수정 | 삭제 복구 |
|  | 퇴사/휴직/복직 처리 |
|  | 승진 처리 |
|  | 부서 변경 |
|  | 이메일 변경 |
|  | 입사일 변경 |
|  | 직급 변경 |
|  | 고용형태 변경 |
|  | 등급 변경 |
|  | 메모 변경 |
|  | 직원 엑셀 업로드 |

즉 `SELF` 쓰기 권한은 `내 프로필 일부 수정` 정도로 이해하면 됩니다.

---

## 6. 직원 목록과 상세 정보의 차이

운영 관점에서 가장 중요한 구분은 `직원 목록`과 `직원 상세`가 다르다는 점입니다.

| 구분 | 설명 |
|---|---|
| 직원 목록 | 로그인한 사용자가 조직 구성원을 확인할 수 있는 공개 디렉터리 |
| 직원 상세 | 민감하거나 상세한 직원 정보가 포함될 수 있는 화면 |

현재 직원 목록에서는 아래 필드를 노출하지 않습니다.

| 목록에서 제외되는 필드 |
|---|
| 생년월일 (`birthDate`) |
| 메모 (`memo`) |

따라서 목록은 비교적 넓게 공개하고, 상세 화면은 권한으로 보호하는 정책을 사용합니다.

## 7. 엑셀 정책

엑셀 기능은 일반 화면 조회보다 더 엄격하게 운영합니다.

| 기능 | 권한 정책 | 설명 |
|---|---|---|
| 직원 엑셀 다운로드 | `employee.read` 필요 | 허용 범위 직원만 다운로드 가능 |
| 직원 엑셀 업로드 | `employee.write` 필요 | 업로드 행 전체가 허용 범위 안에 있어야 함 |
| 직원 엑셀 샘플 다운로드 | 로그인 사용자 사용 가능 | 업로드 양식 제공용 |

특히 엑셀 업로드는 여러 명의 직원을 한 번에 생성하는 기능이므로, 한 행이라도 권한 범위를 벗어나면 전체 요청을 거부합니다.

---

## 8. 기본 그룹 기준

현재 기본 시드는 아래와 같이 구성되어 있습니다.

| 그룹 | `employee.read` | `employee.write` | `permission.group.manage` | 설명 |
|---|---|---|---|---|
| 일반 그룹 | `SELF` | `SELF` | 없음 | 자기 상세 조회, 자기 프로필 일부 수정 가능 |
| 최고 관리자 그룹 | `ALL` | `ALL` | `ALL` | 전 직원 조회 및 관리, 권한 그룹 관리 가능 |

또한 기본 계정 매핑은 아래와 같습니다.

| 계정 | 그룹 |
|---|---|
| 1번 계정 | 최고 관리자 그룹 |
| 2번 계정 | 일반 그룹 |

## 9. 운영자가 이해하면 좋은 한 줄 정리

| 질문 | 답변 |
|---|---|
| 직원 명단은 누가 볼 수 있나요? | 로그인한 사용자는 볼 수 있습니다. |
| 직원 상세는 누가 볼 수 있나요? | `employee.read` 권한이 있는 사용자만 볼 수 있습니다. |
| 직원 정보는 누가 바꿀 수 있나요? | `employee.write` 권한이 있는 사용자만 바꿀 수 있습니다. |
| 일반 직원도 자기 정보 수정이 가능한가요? | 네. 하지만 이름, 생년월일, 아바타만 가능합니다. |
| 관리자만 가능한 작업은 무엇인가요? | 다른 직원의 상세 조회, 조직 단위 엑셀 처리, 직원 생성/삭제/상태 변경 등입니다. |
| 권한 그룹 관리는 누가 할 수 있나요? | `permission.group.manage` 권한이 있는 최고 관리자만 할 수 있습니다. |

---

## 10. 운영상 권장 해석

- 일반 직원: 자기 정보 확인 및 자기 프로필 일부 수정
- 부서 관리자: 자기 부서 또는 하위 부서 기준으로 직원 조회/관리
- 최고 관리자: 전사 직원 조회/관리

이 문서는 현재 구현 기준이며, 이후 `project.read`, `dashboard.read` 등의 실제 적용 범위가 확장되면 함께 업데이트해야 합니다.
