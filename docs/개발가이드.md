# Backend Development Guide (Hexagonal Architecture)

이 가이드는 **Hexagonal Architecture (Ports and Adapters)** 를 기반으로 한 백엔드 개발 표준입니다.
**철저한 계층 분리**를 통해 유지보수성을 극대화하고, **Null Safety**를 보장하는 것을 목표로 합니다.

---

## 1. 핵심 원칙 (Core Principles)

1.  **의존성 방향 (Dependency Rule)**
    *   `adapter` $\rightarrow$ `application` $\rightarrow$ `domain`
    *   **DTO 분리 원칙:** Web 계층의 `Request` 객체는 절대 Application 계층으로 넘어가지 않습니다. 반드시 `Command` 객체로 변환되어야 합니다.
2.  **패키지 구조 (Ports & Adapters)**
    *   **최상위:** 레이어(Layer) 구분 (`domain`, `application`, `adapter`)
    *   **하위:** 도메인(Domain) 구분 (`employee`, `department` 등)
    *   **Inbound/Outbound:** Application 계층은 `inbound` (Use Case/Interface), `outbound` (Port/Interface)로 포트를 명확히 구분합니다.
3.  **Null Safety 전략 (JSpecify)**
    *   모든 패키지에 `package-info.java`를 통해 **`@NullMarked`**를 적용합니다.
    *   기본적으로 **모든 변수와 반환값은 `Non-null`**입니다.
    *   **Service 계층 내부에서는 방어적 로직(`if (obj == null)`)을 작성하지 않습니다.**
4.  **CQRS 패턴 적용**
    *   명령(Command/Write)과 조회(Query/Read)의 모델과 포트를 분리합니다.
    *   **Write:** `Manager` (Interface) $\rightarrow$ `Service` (Implementation)
    *   **Read:** `Finder` (Interface) $\rightarrow$ `Service` (Implementation)

## 3. 데이터 흐름 및 네이밍 (Data Flow & Naming)

**CQRS 원칙에 따라 "상태 변경(Write)"과 "데이터 조회(Read)"의 흐름을 철저히 분리합니다.**

### 3-1. Write Flow (등록, 수정, 삭제, 상태 변경)

**"Request를 받아 Command로 변환하고, Entity를 조작한 후, ID를 반환한다."**

| 단계 | 위치 | 객체명/역할 | 역할 및 특징 |
| :--- | :--- | :--- | :--- |
| **1. 요청** | Adapter (`api`) | `~Request` | **JSON Body.** `@Valid` 검증 수행. (예: `EmployeeCreateRequest`) |
| **2. 변환** | Adapter (`api`) | `toCommand()` | Web 의존성 없는 순수 데이터 객체로 변환. |
| **3. 입력** | Application (`inbound`) | `~Command` | **Manager(Write Port)의 Input.** 불변 객체 (`record`). |
| **4. 포트** | Application (`inbound`) | `~Manager` | **Interface.** 비즈니스 행위(Use Case) 정의. (예: `EmployeeManager`) |
| **5. 처리** | Application | `~Service` | **Implementation.** Entity 로딩/저장, 도메인 로직 실행. (예: `EmployeeModifyService`) |
| **6. 결과** | Application | `Long id` | 명령의 결과로 보통 식별자(ID)만 반환. |
| **7. 응답** | Adapter (`api`) | `~Response` | ID를 감싸서 표준 포맷으로 응답. (예: `EmployeeCreateResponse`) |

### 3-2. Read Flow (조회, 검색)

**"검색 조건(Condition)을 받아, DB에서 최적화된 DTO(Summary/Detail)를 조회한다."**

| 단계 | 위치 | 객체명/역할 | 역할 및 특징 |
| :--- | :--- | :--- | :--- |
| **1. 요청** | Adapter (`api`) | `~Request` | **Query Params.** (예: `EmployeeSearchCondition` 직접 바인딩) |
| **2. 포트** | Application (`inbound`) | `~Finder` | **Interface.** 조회 전용 Use Case 정의. (예: `EmployeeFinder`) |
| **3. 입력** | Application (`inbound`) | `~Condition` | **Finder(Read Port)의 Input.** 검색 조건. |
| **4. 처리** | Application | `~QueryService` | **Implementation.** Repository를 통해 DB에서 DTO 직접 조회. |
| **5. 결과** | Application | `~Summary` / `~Detail` | **목록:** 가벼운 `Summary` <br>**단건:** 상세한 `Detail` |
| **6. 응답** | Adapter (`api`) | `~Response` | API 스펙에 맞춰 데이터를 감싸서 응답. |

## 4. Null Safety 구현 가이드

### 4-1. 패키지 설정 (`package-info.java`)

각 패키지 폴더마다 아래 파일을 생성하여, 해당 패키지 내 모든 코드가 기본적으로 **Null을 허용하지 않음**을 선언합니다.

```java
@NullMarked
package kr.co.abacus.abms.domain.employee;

import org.jspecify.annotations.NullMarked;
```

### 4-2. 코드 작성 규칙

*   **기본:** 아무 어노테이션이 없으면 `Non-null`입니다.
*   **예외:** Null이 들어올 수 있는 경우에만 `@Nullable` (`org.jspecify.annotations.Nullable`)을 명시합니다.

## 5. 구현 가이드 (Employee 예시)

### 5-1. [Inbound Port] Use Case Interface

**Write (Manager)**: `application/employee/inbound/EmployeeManager.java`

```java
public interface EmployeeManager {
    Long create(EmployeeCreateCommand command);
    Long updateInfo(Long id, EmployeeUpdateCommand command);
    void promote(Long id, EmployeePosition newPosition, @Nullable EmployeeGrade newGrade);
    // ...
}
```

**Read (Finder)**: `application/employee/inbound/EmployeeFinder.java`

```java
public interface EmployeeFinder {
    EmployeeDetail findEmployeeDetail(Long id);
    Page<EmployeeSummary> search(EmployeeSearchCondition condition, Pageable pageable);
}
```

### 5-2. [Application Service] Implementation

**Write Service**: `application/employee/EmployeeModifyService.java`

```java
@Service
@Transactional
@RequiredArgsConstructor
public class EmployeeModifyService implements EmployeeManager {
    private final EmployeeRepository employeeRepository;

    @Override
    public Long create(EmployeeCreateCommand command) {
        // 1. 도메인 로직 검증 (중복 등)
        validateDuplicateEmail(command.email());

        // 2. Entity 생성
        Employee employee = command.toEntity();

        // 3. 저장
        return employeeRepository.save(employee).getId();
    }
}
```

### 5-3. [Outbound Port] Repository Interface

`application/employee/outbound/EmployeeRepository.java`

```java
public interface EmployeeRepository extends Repository<Employee, Long> {
    Employee save(Employee employee);
    Optional<Employee> findById(Long id);
    boolean existsByEmail(Email email);
    
    // Custom Repository 기능 확장 (QueryDSL 등)
    // CustomEmployeeRepository 상속
}
```

### 5-4. [Adapter] Web Controller

`adapter/api/employee/EmployeeApi.java`

```java
@RestController
@RequiredArgsConstructor
public class EmployeeApi {
    private final EmployeeManager employeeManager; // Inbound Port 주입
    private final EmployeeFinder employeeFinder;   // Inbound Port 주입

    @PostMapping("/api/employees")
    public EmployeeCreateResponse create(@RequestBody @Valid EmployeeCreateRequest request) {
        // Request -> Command 변환 후 Manager 호출
        Long id = employeeManager.create(request.toCommand());
        return EmployeeCreateResponse.of(id);
    }

    @GetMapping("/api/employees")
    public PageResponse<EmployeeSearchResponse> search(
            @Valid EmployeeSearchCondition condition, 
            Pageable pageable
    ) {
        // Finder 호출 -> Service -> Repository -> DB
        Page<EmployeeSummary> result = employeeFinder.search(condition, pageable);
        
        // Summary -> Response 변환
        return PageResponse.of(result.map(EmployeeSearchResponse::of));
    }
}
```

### 5-5. [Adapter] Infrastructure (QueryDSL)

`adapter/infrastructure/employee/EmployeeRepositoryImpl.java`

*   QueryDSL을 사용하여 복잡한 조회 쿼리를 작성합니다.
*   엔티티를 조회하는 것이 아니라, `Projections.constructor`를 사용하여 필요한 DTO(`EmployeeSummary` 등)를 바로 생성합니다.
