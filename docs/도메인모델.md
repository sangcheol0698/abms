# ABMS 도메인 모델

본 문서는 현재 코드에 반영된 활성 도메인 모델을 요약합니다.
기준은 `abms-domain` 모듈이며, 준비 중인 개념은 별도 표시합니다.

## 1. 공통 기반

### 1.1 AbstractEntity

- `id`: `Long`
- `BaseEntity`를 상속하는 모든 엔티티의 공통 식별자입니다.

### 1.2 BaseEntity

- `createdAt`: `LocalDateTime`
- `updatedAt`: `LocalDateTime`
- `createdBy`: `Long`
- `updatedBy`: `Long`
- `deleted`: `Boolean`
- `deletedAt`: `LocalDateTime?`
- `deletedBy`: `Long?`

### 1.3 공통 값 객체

#### Email

- `address`: `String`
- 이메일 형식을 검증합니다.

#### Period

- `startDate`: `LocalDate`
- `endDate`: `LocalDate?`
- 시작일은 종료일보다 늦을 수 없습니다.

#### Money

- `amount`: `BigDecimal`
- `wons()`, `zero()`, `add()`, `subtract()`, `multiply()`, `divide()`를 제공합니다.

## 2. 직원/조직

### 2.1 Employee

직원 Aggregate Root입니다.

#### 주요 필드

- `departmentId`: `Long`
- `name`: `String`
- `email`: `Email`
- `joinDate`: `LocalDate`
- `birthDate`: `LocalDate`
- `position`: `EmployeePosition`
- `type`: `EmployeeType`
- `status`: `EmployeeStatus`
- `grade`: `EmployeeGrade`
- `avatar`: `EmployeeAvatar`
- `resignationDate`: `LocalDate?`
- `memo`: `String?`

#### 주요 행위

- `create()`
- `resign()`
- `takeLeave()`
- `activate()`
- `updateInfo()`
- `promote()`
- `softDelete()`
- `restore()`

#### 규칙

- 생성 시 상태는 `ACTIVE`
- 퇴사일은 입사일 이후여야 함
- 퇴사자는 수정 및 승진 불가
- 삭제 시 이메일을 `deleted.{timestamp}.{original}` 형식으로 마스킹

### 2.2 EmployeePosition

- `ASSOCIATE`, `SENIOR_ASSOCIATE`, `PRINCIPAL`, `TEAM_LEADER`, `CHIEF`
- `DIRECTOR`, `TECHNICAL_DIRECTOR`, `MANAGING_DIRECTOR`, `VICE_PRESIDENT`, `PRESIDENT`, `CHAIRMAN`

### 2.3 EmployeeType

- `FULL_TIME`, `FREELANCER`, `OUTSOURCING`, `PART_TIME`

### 2.4 EmployeeGrade

- `JUNIOR`, `MID_LEVEL`, `SENIOR`, `EXPERT`

### 2.5 EmployeeStatus

- `ACTIVE`, `ON_LEAVE`, `RESIGNED`

### 2.6 EmployeeAvatar

- UI 프로필용 아바타 프리셋 enum입니다.

### 2.7 Department

부서 Aggregate Root입니다.

#### 주요 필드

- `code`: `String`
- `name`: `String`
- `type`: `DepartmentType`
- `leaderEmployeeId`: `Long?`
- `parent`: `Department?`
- `children`: `List<Department>`

#### 주요 행위

- `create()`
- `assignLeader()`

#### 규칙

- 부서 코드는 고유
- 부모-자식 관계를 양방향으로 유지

### 2.8 DepartmentType

- `COMPANY`, `DIVISION`, `GROUP`, `TEAM`, `LAB`, `TF`

### 2.9 PositionHistory

직원의 직급 변경 이력을 저장하는 엔티티입니다.

- `employeeId`: `Long`
- `period`: `Period`
- `position`: `EmployeePosition`
- `create()`

### 2.10 Payroll

직원의 급여 이력 엔티티입니다.

- `employeeId`: `Long`
- `annualSalary`: `Money`
- `period`: `Period`
- `create()`
- `close()`

### 3.3 ProjectStatus

### 3.4 ProjectRevenuePlan

프로젝트 하위의 매출 계획 엔티티입니다.

#### 주요 필드

- `projectId`: `Long`
- `sequence`: `Integer`
- `revenueDate`: `LocalDate`
- `type`: `RevenueType`
- `amount`: `Money`
- `isIssued`: `Boolean`
- `memo`: `String?`

#### 주요 행위

- `create()`
- `issue()`
- `cancel()`

#### 규칙

- 같은 프로젝트에서 `sequence` 중복 불가
- 매출은 `revenueDate`와 `isIssued`를 기준으로 집계

### 3.5 RevenueType

- `DOWN_PAYMENT`, `INTERMEDIATE_PAYMENT`, `BALANCE_PAYMENT`, `MAINTENANCE`, `ETC`

### 3.6 ProjectAssignment

프로젝트 투입 엔티티입니다.

#### 주요 필드

- `projectId`: `Long`
- `employeeId`: `Long`
- `role`: `AssignmentRole?`
- `period`: `Period`

#### 주요 행위

- `assign()`
- `calculateManMonth()`

#### 규칙

- 투입 시작일은 프로젝트 시작일보다 빠를 수 없음
- 프로젝트 종료일이 있으면 투입 종료일은 그보다 늦을 수 없음
- MM은 월 총일수 대비 실제 투입일수로 계산

### 3.7 AssignmentRole

- `DEV`, `PM`, `PL`, `ETC`

### 3.8 EmployeeCostPolicy

직원 유형별 월 원가 계산 정책입니다.

#### 주요 필드

- `applyYear`: `Integer`

- `overheadRate`: `Double`
- `sgaRate`: `Double`

#### 주요 행위

- `calculateEmployeeCost()`

### 3.9 EmployeeMonthlyCost

### 3.10 MonthlyRevenueSummary

프로젝트별 월별 손익 집계 엔티티입니다.

- `projectId`: `Long`
- `projectCode`: `String`
- `projectName`: `String`
- `leadDepartmentId`: `Long`

- `leadDepartmentName`: `String`

- `create()`

## 6. 준비 중 개념

### Contract

- `Contract`와 `ContractType`은 도메인 코드는 존재하지만 현재 `@Entity`가 비활성화되어 있습니다.
- 운영 중인 활성 모델로 보기보다 향후 계약 버전 관리 확장용 준비 영역으로 해석하는 편이 맞습니다.
