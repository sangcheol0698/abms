
본 문서는 현재 ABMS 프로젝트의 인증(Authentication) 구현을 기반으로,  
**다른 프로젝트에서도 재사용 가능한 형태**로 인증 아키텍처와 구현 흐름을 정리한 가이드입니다.

- 대상: 백엔드(Spring Security) + 프론트엔드(Vue) 기반 웹 애플리케이션
- 범위: 로그인/로그아웃, 세션 유지, 현재 사용자 조회, 비밀번호 변경, 가입 요청/확정, CSRF 초기화
- 비범위: 세부 인가(Authorization) 정책(권한 코드/스코프 매트릭스)은 `권한가이드.md` 참고


## 1. 한눈에 보는 아키텍처

현재 인증 구조는 **계층 분리 + 포트/어댑터(헥사고날)** 를 따릅니다.

1. API 계층(웹 어댑터)
   - HTTP 요청/응답 처리
   - 입력 검증
   - 애플리케이션 서비스 호출
   - 로그인 성공 시 Security Context 저장

2. 애플리케이션 계층
   - 인증 유스케이스 조합(로그인, 가입, 비밀번호 변경)
   - 트랜잭션 경계
   - 도메인 예외를 통한 실패 케이스 관리

3. 인프라/시큐리티 어댑터
   - `UserDetailsService`, `AuthenticationManager` 연동
   - 패스워드 인코딩
   - Security Context 기반 감사(auditing) 지원

4. 프론트엔드 인증 클라이언트
   - 로그인/로그아웃/세션 동기화
   - `me` 조회 기반 사용자 상태(localStorage + query cache) 유지
   - 포커스/가시성 변화 시 서버 세션 재검증



## 2. 핵심 설계 원칙

### 2.1 인증과 인가의 분리
- 인증은 “누구인가”를 확인하는 흐름에 집중합니다.
- 인가는 “무엇을 할 수 있는가”를 권한 서비스로 분리합니다.
- 결과적으로 `AuthQueryService`는 사용자 기본 정보 + 권한 목록을 조합해 반환하지만, 권한 판정 로직 자체는 별도 책임에 둡니다.

### 2.2 상태 기반 세션 인증
- 로그인 성공 후 `SecurityContext`를 저장합니다.
- 이후 요청은 서버 세션 컨텍스트를 기반으로 현재 사용자(`me`)를 조회합니다.
- JWT를 직접 다루지 않고, 서버 세션 + CSRF 패턴으로 웹 보안 모델을 구성합니다.

### 2.3 포트/어댑터를 통한 기술 의존성 격리
- 애플리케이션 서비스는 `CredentialAuthenticator`, `AccountRepository` 등 포트에만 의존합니다.
- Spring Security, JPA 구현은 어댑터에서 담당합니다.
- 다른 프로젝트에서 구현체만 교체하면 동일한 유스케이스를 재사용할 수 있습니다.



## 3. 백엔드 구현 가이드

### 3.1 Security 코어 설정
`SecurityCoreConfig`에서 기본 인증 컴포넌트를 제공합니다.

- `PasswordEncoder`: Delegating encoder 사용
- `AuthenticationManager`: `DaoAuthenticationProvider` 기반 구성
- `UserDetailsService` + `PasswordEncoder`를 연결해 username/password 인증 수행

핵심 포인트:
- 프로젝트 전역에서 별도 Bean이 없으면 기본값으로 동작
- 테스트/특수 환경에서 오버라이드가 쉬운 구조

### 3.2 사용자 조회 어댑터
`CustomUserDetailsService`:
- username(이메일)로 계정 조회
- 없으면 `UsernameNotFoundException`
- 있으면 `CustomUserDetails`로 변환

`CustomUserDetails`:
- 최소한의 인증 정보 보관: `accountId`, `username`, `password`
- 현재 구현에서 `authorities`는 빈 목록
  - 권한 체크는 별도 권한 도메인/서비스에서 처리하는 방향


### 3.3 로그인 처리 흐름

`AuthApi.login`:
1. `POST /api/auth/login` 요청 수신
2. `AuthManager.login()` 호출
3. 성공 시 `SecurityContextRepository.saveContext(...)` 호출로 컨텍스트 저장

`AuthCommandService.login`:
1. `CredentialAuthenticator.authenticate(username, password)` 호출
2. 구현체(`SpringSecurityCredentialAuthenticator`)에서
   - `AuthenticationManager.authenticate(...)`
   - 성공한 `Authentication`을 `SecurityContextHolder`에 주입

핵심 포인트:
- 애플리케이션 서비스는 “인증 성공/실패” 유스케이스만 표현
- 실제 Spring Security 객체 조작은 어댑터가 수행



### 3.4 현재 사용자 조회(`me`)

`AuthApi.me`:
- `GET /api/auth/me`
- `Authentication`에서 username 추출 후 `authFinder.getCurrentUser(...)` 호출

`AuthQueryService.getCurrentUser`:
1. username으로 `Account` 조회(삭제 계정 제외)
2. 연계된 `Employee`가 있으면 사용자 정보 + 권한 목록 반환
3. 없으면 fallback 사용자 정보(로컬 파트 기반 이름, 빈 권한) 반환

핵심 포인트:
- 운영 중 데이터 불일치(계정은 있는데 직원 정보 누락)에 대비한 fallback 전략
- 프론트가 최소한의 사용자 표시는 유지 가능

### 3.5 비밀번호 변경

`PATCH /api/auth/password`:
1. 현재 로그인 username을 기준으로 계정 조회
2. 현재 비밀번호 검증 실패 시 예외
3. 신규 비밀번호가 기존과 같으면 예외
4. 인코딩 후 계정 비밀번호 변경

권장사항(다른 프로젝트 적용 시):
- 비밀번호 정책(길이/복잡도/재사용 제한) 검증기를 분리해 주입
- 변경 이벤트 감사 로그(누가, 언제, IP) 저장

### 3.6 비밀번호 찾기/재설정

### 재설정 요청
`POST /api/auth/password-reset-requests`
1. 회사 이메일 형식 검증
2. 활성 계정이 있으면 30분 만료 재설정 토큰 생성
3. 기존 재설정 토큰 삭제 후 새 토큰 저장
4. 재설정 링크 메일 발송
5. 계정 존재 여부와 무관하게 외부 응답은 성공으로 통일

### 재설정 확정
`POST /api/auth/password-reset-confirmations`
1. 토큰 조회 및 유효성 검증(만료/사용 처리)
2. 계정 조회
3. 신규 비밀번호가 기존과 같으면 예외
4. 인코딩 후 계정 비밀번호 변경
5. 사용된 토큰 삭제

핵심 포인트:
- 계정 존재 여부를 응답으로 노출하지 않아 이메일 열거를 방지합니다.
- 가입 토큰과 별도 엔티티를 사용해 목적, 메일 문구, 만료/사용 이력을 분리합니다.
- 비밀번호 정책은 회원가입/비밀번호 변경과 동일하게 유지합니다.

### 3.7 가입 요청/가입 확정(이메일 토큰 방식)

### 가입 요청
`POST /api/auth/registration-requests`
1. 이메일 중복 계정 여부 확인
2. 가입 가능한 직원인지 검증
3. 30분 만료 토큰 생성
4. 기존 토큰 삭제 후 새 토큰 저장
5. 메일 발송(실패 시 도메인 예외로 전환)

### 가입 확정
`POST /api/auth/registration-confirmations`
1. 토큰 조회 및 유효성 검증(만료/사용 처리)
2. 중복 계정 재검증
3. 계정 생성(비밀번호 인코딩)
4. 기본 권한 그룹 자동 할당
5. 사용된 토큰 삭제

핵심 포인트:
- “요청”과 “확정” 단계를 분리해 계정 탈취 리스크 감소
- 토큰 단건 사용(consumed) 패턴으로 재사용 방지
- 가입 직후 기본 권한 그룹 부여로 초기 접근 정책 일관성 확보

### 3.8 CSRF 초기화 엔드포인트

`GET /api/csrf`:
- 빈 `200 OK` 응답
- 프론트가 초기 진입 시 CSRF 쿠키/토큰 컨텍스트를 안정적으로 준비하는 용도



## 4. 프론트엔드 구현 가이드

### 4.1 인증 쿼리/뮤테이션 계층

`useAuthQueries`는 인증 API를 일관된 인터페이스로 제공합니다.

- `useAuthMeQuery`
- `useLoginMutation`
- `useLogoutMutation`
- `useChangePasswordMutation`
- `useRequestRegistrationMutation`
- `useConfirmRegistrationMutation`
- `useRequestPasswordResetMutation`
- `useConfirmPasswordResetMutation`

특징:
- 로그인 성공 시 `authKeys.me()` invalidate
- 로그아웃 성공 시 인증 관련 query cache 제거

### 4.2 세션 저장소 전략

`session.ts` 핵심 동작:
- 사용자 정보(localStorage) 저장/파싱/정규화
- 권한 코드/스코프 조회 유틸 제공
- 세션 검증 상태 플래그(`isSessionValidated`) 관리
- 서버 `me` 재조회로 세션 유효성 확인

주요 함수:
- `setStoredUser`, `getStoredUser`, `clearStoredUser`
- `ensureServerSessionValid(forceRefresh)`
- `refreshStoredUserSession(force)`
- `initializeSessionSync()`

### 4.3 포커스/가시성 기반 세션 동기화

브라우저 포커스 획득 또는 탭 visibility 변경 시:
1. 로컬에 사용자 정보가 있으면
2. 일정 간격 기준으로 `me` 재검증
3. 실패 시 로컬 사용자 정보 및 CSRF 상태 정리

효과:
- 서버 세션 만료/무효화 시 UI 상태가 자동으로 정합성 회복
- 새로고침 없이도 인증 상태를 비교적 안정적으로 유지



## 5. 재사용 가능한 API 계약 템플릿

다른 프로젝트에서도 아래 계약을 기본 템플릿으로 권장합니다.

- `POST /api/auth/login`
  - request: `{ username, password }`
  - response: `200/204`
- `POST /api/auth/logout` (또는 시큐리티 기본 로그아웃 경로)
- `GET /api/auth/me`
  - response: `{ name, email, employeeId?, departmentId?, permissions[] }`
- `PATCH /api/auth/password`
  - request: `{ currentPassword, newPassword }`
- `POST /api/auth/password-reset-requests`
  - request: `{ email }`
- `POST /api/auth/password-reset-confirmations`
  - request: `{ token, password }`
- `POST /api/auth/registration-requests`
  - request: `{ email }`
- `POST /api/auth/registration-confirmations`
  - request: `{ token, password }`
- `GET /api/csrf`



## 6. 예외/에러 처리 권장 규칙

1. 도메인 의미가 있는 예외명을 사용
   - 예: `InvalidCurrentPasswordException`, `AccountAlreadyExistsException`
2. 사용자 메시지와 내부 원인을 분리
   - 사용자: 안전하고 이해 가능한 메시지
   - 내부: 로그/모니터링용 상세 원인
3. 인증 실패와 시스템 실패를 구분
   - 인증 실패(4xx), 인프라 실패(5xx)



## 7. 보안 체크리스트 (다른 프로젝트 적용용)

1. 비밀번호
   - 강한 해시 알고리즘/Delegating encoder
   - 최소 길이/복잡도 정책
   - 동일 비밀번호 재사용 방지

2. 세션
   - 로그인/권한변경 시 세션 재발급 여부 검토
   - 만료 시간, 동시 세션 정책 정의
   - 로그아웃 시 서버 세션 무효화 확인

3. CSRF
   - 상태 변경 요청(POST/PATCH/DELETE)에 CSRF 보호 적용
   - 프론트 초기화 엔드포인트 또는 첫 페이지 로드에서 토큰 준비

4. 계정 가입
   - 토큰 만료 시간 짧게 유지
   - 토큰 단회성 소비
   - 메일 발송 실패 재시도/관측 가능성 확보

5. 관측성
   - 로그인 성공/실패, 비밀번호 변경, 가입 확정 이벤트 로깅
   - 이상 징후(반복 실패, 비정상 IP) 알림



## 8. ABMS 기준 구현 매핑(코드 이해용)

- 인증 API 진입점: `abms-adapter-web/.../auth/AuthApi`
- CSRF 초기화 API: `abms-adapter-web/.../auth/CsrfApi`
- 인증 유스케이스 서비스: `abms-application/.../auth/AuthCommandService`
- 인증 조회 서비스: `abms-application/.../auth/AuthQueryService`
- 시큐리티 설정: `abms-adapter-integration/.../security/config/SecurityCoreConfig`
- UserDetails: `abms-adapter-integration/.../security/CustomUserDetails`
- UserDetailsService: `abms-adapter-integration/.../security/service/CustomUserDetailsService`
- 인증 수행 어댑터: `abms-adapter-integration/.../security/service/SpringSecurityCredentialAuthenticator`
- 프론트 인증 훅: `frontend/src/features/auth/queries/useAuthQueries.ts`
- 프론트 세션 관리: `frontend/src/features/auth/session.ts`



## 9. 빠른 도입 순서 (신규 프로젝트용)

1. 백엔드
   - `UserDetailsService`, `PasswordEncoder`, `AuthenticationManager` 구성
   - `login`, `me`, `logout`, `change-password` API부터 구현

2. 프론트
   - 로그인/로그아웃 + `me` 조회 + localStorage 동기화 구현
   - 라우터 가드에서 `ensureServerSessionValid` 연동

3. 확장
   - 가입 요청/확정(메일 토큰) 추가
   - 권한 그룹/스코프 모델 연결
   - 감사 로그/보안 모니터링 추가



## 10. 결론

ABMS의 인증 구현은 다음 장점이 있습니다.

- 세션 기반 웹 보안 모델에 충실
- 애플리케이션 계층과 시큐리티 인프라의 책임 분리
- fallback/세션 동기화 등 운영 현실을 고려한 안정성 확보
- 다른 프로젝트로 이식 가능한 포트/어댑터 구조

즉, 이 문서를 기준으로 하면  
**“기능이 동작하는 인증”을 넘어서 “운영 가능한 인증”을 설계/구현**할 수 있습니다.
